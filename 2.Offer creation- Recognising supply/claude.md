# Offer Creation OS — Claude.md

## Core Rule: Build Skills, Not Products
The game is to package founder productivity expertise as reusable, AI-callable skills.
(Greg Isenberg's Skill Era framework)

The full stack:

```
CLI  = where the founder works (Claude Code terminal)
MCP  = the connector that plugs your skill into their CLI
Skill = your judgment (debt diagnosis — why the founder is stuck)
API  = the mechanical tools underneath (fetch data, manage tasks)
```

An API is a doorway into a function.
A skill is a doorway into judgment.
MCP is the plug that makes the skill callable.
CLI is the surface where the founder experiences it.

Your game is building the skill. CLI and MCP are delivery mechanisms — they may evolve or replace each other. The skill stays the same regardless of how it's delivered.

## What This Means for mywellnessgame
- Your 3-Game Framework, debt diagnosis, and tollgate system are not just methodology — they are potential skills
- Document your judgment process (how you diagnose, decide, and message) — this is the distributable IP
- The barrier to building software is lowering; the barrier to encoding real-world expertise is rising

## CLI vs API vs Skill vs MCP

```
CLI  = the surface (interface — where the founder types and works)
API  = the tool (function — "here's how to manage your to-do list")
Skill = the judgment (diagnosis — "here's why your to-do list drains you")
MCP  = the connector (delivery — lets a founder's AI call your skill while they work)
```

**CLI** — the terminal where the founder works (Claude Code). This is where they're staring at their task list, feeling stuck.
**API** — any productivity app does this (Notion, Todoist). Manages tasks mechanically.
**Skill** — only your framework does this. Diagnoses why a founder freezes at their task list. Context Debt, Content Debt, or Formula Debt.
**MCP** — the plumbing that lets AI agents access your skill. You don't build this. You build the judgment that flows through it.

Your game is building skills. CLI is where founders experience it. MCP is how it gets delivered. API is the mechanical layer underneath.
Founders setting up Claude Code for non-technical work are already in the CLI — they just don't have the skill plugged in yet.

## First Validated Skill — Macro → Micro → Nano Game Visibility

This skill is validated. Sujith (paying client) used it and said: "the experience of doing this is wild" and "it is building courage."

The skill makes the game visible. Visibility builds courage. Regulation happens before the coaching conversation.

### The 11 Template Columns

| # | Column | Question / Purpose |
|---|---|---|
| 1 | Date | Date of entry |
| 2 | Type of leverage | High / Medium / Low |
| 3 | Macro game | Based on your current situation, what is the Macro level — high leverage outcome that one needs to accomplish to move in the direction of neutralising the situation that is coming in the way of making our work situation simpler to engage? Note: Achieving this usually takes 6–12 weeks. |
| 4 | Micro game | What is the Micro level — high leverage outcome that one needs to accomplish to move in the direction of accomplishing macro level — high leverage outcome? Note: Achieving this usually takes 1–6 weeks. |
| 5 | Nano game | What is the nano level — high leverage outcomes that one needs to accomplish to move in the direction of accomplishing micro and/or macro level — high leverage outcomes? Note: Time that it takes to accomplish can be 1–3 weeks. |
| 6 | Constraints — Game type | Constraints in the NANO game wrt three types of credits game: Clarity / Action / Lesson |
| 7 | Constraints — Sub credits | Constraints in the NANO game wrt two types of sub credits inside each credit: Clarity (concepts or qualities), Action (tasks or tactics), Lesson (workflows or capabilities) |
| 8 | Next steps | Next steps for nano game — during the day |
| 9 | Assessment status | Status for assessment being done in remember sheet (Yes/No) |
| 10 | Journal | Covers mostly NANO — and if micro and/or macro moves, mention them |
| 11 | Nano game progress | Complete or Incomplete |

### What this skill can and cannot do

**Can do:**
- Map any situation to Macro → Micro → Nano
- Identify the Game at Nano level (Clarity / Action / Lesson)
- Surface the sub-credit constraint (concepts/qualities, tasks/tactics, workflows/capabilities)
- Generate next steps for the day
- Create felt regulation — the founder sees what is breaking, not just experiences it

**Cannot do without the coach:**
- Regulate the founder when overwhelmed
- Hold them to the ONE thing over time
- Surface patterns across weeks
- Diagnose which debt stage the founder is actually in

## The Coach Role — How You Show Up

| Lens | Founder Shares | Coach Role | Game |
|---|---|---|---|
| Recognise | To-do list | Thinking Partner | Macro (Context Debt) |
| Remember | Assessment | Accountability Partner | Micro (Content Debt) |
| Review | Journal | Coaching Partner | Nano (Formula Debt) |

**Recognise** — founder shares their to-do list. Coach is a thinking partner. Helps them see what matters right now.
**Remember** — founder shares their assessment. Coach is an accountability partner. Holds them to the one thing.
**Review** — founder shares their journal. Coach is a coaching partner. Makes invisible patterns visible.

The diagnostic (demand side) tells you where the founder is stuck.
The coach role (offer side) tells you how you show up to help them move.
